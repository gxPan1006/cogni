# 20260516_185318

## Summary

Dogfood basic infra:cloud 加 `/auth/dev-token` HTTP 端点,desktop useAuth 加 vite-dev-mode fallback,自动 fetch dev token。一次写,以后所有 dogfood 都不需要再手动 inject token 或走 Safari Web Inspector。

## Context

补完 Phase 3 整合后,用户实测 dogfood 卡在登录:Google OAuth 在用户网络环境不通,之前用 `mint-dev-token.ts` 签出的 token 没法注入到 webview localStorage(Safari Inspector 路径用户搞不通),`open cogni://auth?token=...` 因 LSRegister 指向旧 Cogni.app bundle 没生效。

## Changes

### cloud:`/auth/dev-token` HTTP 端点

- `packages/cloud/src/routes/auth.ts` —
  - 在 `registerAuthRoutes` 里加 `if (process.env.NODE_ENV !== "production") app.post("/auth/dev-token", ...)`
  - 跟 `scripts/mint-dev-token.ts` 同一逻辑:`findOrCreateUser({ oauthSub: "dev|manual", email: "dev-manual@local.test" })` → `auth.issueToken()` → 返回 `{ token }`
  - **直观表现**:本地 `curl -X POST http://localhost:8787/auth/dev-token` 立刻拿到一个签好的 JWT,**生产构建不会注册此路由**

### cloud:CORS 扩到 `/auth/*`

- `packages/cloud/src/server.ts` —
  - 提取 `corsMiddleware` 变量,挂到 `/api/*` 和 `/auth/*` 两个前缀
  - 之前只在 `/api/*` 上挂 → 桌面 webview 跨域 fetch `/auth/dev-token` 会被浏览器拦
  - `/auth/google/start` 和 `/auth/google/callback` 是浏览器 redirect 不是 XHR,CORS 对它们是 no-op,白挂无害

### desktop:useAuth 加 dev fallback

- `apps/desktop/src/useAuth.ts` —
  - 新增一个 `useEffect`(depends `[token]`):
    - `if (!import.meta.env.DEV) return;` — vite build 时 dead-code-eliminated,生产 bundle 不含此分支
    - `if (token) return;` — 已经有 token 就不动
    - 否则 fetch `/auth/dev-token`,拿到 token 写 localStorage + `setToken`
  - `alive` 标志位防止组件 unmount 后 setState
  - **直观表现**:`pnpm tauri dev` 起来的 Cogni,如果 localStorage 没 token,**自动进 Welcome 页**(免 Google OAuth + 免手动 inject)
  - **生产打包**:`vite build` 把 `import.meta.env.DEV` 替换为字面 `false`,整段 effect 在 tree-shake 时被删,生产用户必须走 Google OAuth

## 注意事项

- `/auth/dev-token` **绝对不要部署到生产**。`NODE_ENV !== "production"` 检查 + vite 的 DEV 双重保险
- dev token 持久化在 localStorage,localStorage 清掉(或换 Tauri 实例)就会重签
- 这跟 `scripts/mint-dev-token.ts` 是冗余的两条路径:script 适合 CLI / CI 场景,HTTP endpoint 适合 desktop dev runtime;两个都保留
